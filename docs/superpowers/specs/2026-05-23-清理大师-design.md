# 清理大师小游戏 — 设计规格(MVP)

> 起草日期:2026-05-23
> 项目类型:微信小游戏(Canvas 2D,竖屏)
> 阶段:MVP(最小可行产品)

## 1. 产品定位

| 维度 | 决策 |
|---|---|
| 品类 | 极简超休闲(Hyper-casual) |
| 清洁对象 | 多品类杂交(鞋 / 衣 / 车 / 地板 / 餐具 / 宠物 / 牙齿 …) |
| 单关玩法 | 3-5 步擦除,顺序执行 |
| 关卡数 | 20 关(MVP 首发) |
| 变现 | MVP 不接广告,先用埋点验证留存 |
| 社交 | 仅做转发卡片(`wx.shareAppMessage`),无后端 |
| 进度持久化 | `wx.setStorage` 本地存储 |
| 数据上报 | 微信云开发 `events` 集合 + `trackEvent` 云函数 |
| 美术 | 阶段一占位图 → 阶段二 AI 生图 |
| 关卡组织 | 纯数据驱动(`data/levels.js` 数组) |

**核心假设:** 多品类换皮 + 3-5 步擦除递进感 → 单局 30-90 秒 → 次日留存可观。先以最低成本上线验证,再决定是否投美术与广告预算。

## 2. 架构总览

```
game.js (入口)
  └─ GameScene (场景控制器:渲染循环 + 触摸路由)
       ├─ LevelRunner    // 关卡控制器:加载配置 → 步骤流转 → 通关回调
       │     └─ DirtLayer × N   // 每个步骤一个污垢层(沿用原引擎)
       ├─ UI             // 顶部进度条 / 工具栏 / Toast / 胜利弹窗
       ├─ storage        // 本地进度读写(wx.setStorage)
       ├─ tracker        // 云开发埋点上报
       └─ share          // 转发卡片
  data/
    ├─ levels.js         // 20 关配置
    └─ tools.js          // 工具元数据
```

**与现有原型的差异:**

- `stages.js` 升级为 `data/levels.js`,从单关多步骤扩展为多关多步骤
- 新增 `LevelRunner`,从 `GameScene` 拆出步骤推进/工具校验/通关检测
- 新增三个边缘模块:`storage` / `tracker` / `share`
- `DirtLayer` 完全保留(经验证可用的擦除引擎)

### 3.3 设计约束

- 关卡只声明它用到的工具(`steps[].tool`),工具栏按当前关动态生成,避免按钮拥挤
- 步骤数 3-5 由数组长度决定,引擎不硬编码
- `layer` 字段完整保留原 `DirtLayer` 接口,引擎不改
- 占位图统一放 `images/placeholder/`,正式资源放 `images/levels/`,后期换图改路径不改字段

## 4. LevelRunner 与游戏流转

### 4.1 职责边界

| 模块 | 管什么 |
|---|---|
| `GameScene` | 渲染循环、触摸事件路由、当前 LevelRunner 生命周期、UI 实例、加载底图 |
| `LevelRunner` | 当前关数据、步骤索引、当前 `DirtLayer`、工具校验、过关检测、步骤推进 |
| `DirtLayer` | 单步骤擦除引擎(原封不动) |

### 4.2 LevelRunner 接口

```js
class LevelRunner {
  constructor(levelData, shoeRect, callbacks)
  // callbacks: { onStepComplete(stepIndex), onLevelComplete(), onWrongTool(expected, got) }

  get currentStep()        // 当前步骤数据
  get overallProgress()    // 0~1,全关合成进度 = (stepIndex + currentStepRatio) / totalSteps
  get stepIndex()          // 当前第几步(0-based)

  setSelectedTool(toolId)
  erase(x, y)              // 工具不对触发 onWrongTool;否则真正擦除
  tick(now)                // 每帧调用,内部判断过关 → 推进步骤
  destroy()
}
```

### 4.3 关卡内流转

```
进入关卡
  → new LevelRunner(level, rect, callbacks)
  → 异步加载 objectImage
  → 加载完后创建第 0 步 DirtLayer + setMask(objectImage)
  → 玩家选工具 → 触摸擦除
  → tick() 发现 progress >= passThreshold
     → 触发 onStepComplete(stepIndex)
     → stepIndex++
     → 销毁旧 DirtLayer,新建下一步 DirtLayer
  → 全部步骤完成
     → 触发 onLevelComplete()
       → GameScene 显示胜利弹窗 + 上报埋点 + 写本地进度
```

### 4.4 关卡间切换

胜利弹窗 → 玩家点 `下一关` → `GameScene` 销毁当前 `LevelRunner` → 用 `levels[N+1]` 新建一个 → 重新走流程。末关只显示 `再玩一次` 与 `分享`。

### 4.5 资源加载策略

- 底图 `objectImage` 用 `wx.createImage()` 异步加载
- 加载期间 UI 显示 loading 文字,避免黑屏
- 加载完成后才创建第 0 步 `DirtLayer`(因污垢层需鞋形蒙版)

## 5. UI 设计

### 5.1 布局(竖屏)

```
┌────────────────────────────┐  ← 顶部 56px,半透明黑底
│ ◀ 返回    关卡 3/20   68% │  ← 单进度条(全关合成进度)
├────────────────────────────┤
│                            │
│       清洁对象画面         │  ← 中央可用区
│                            │
├────────────────────────────┤  ← 底部 100px,半透明黑底
│   [喷雾] [刷子] [抹布]     │  ← 当前关用到的工具(动态生成)
└────────────────────────────┘
```

### 5.2 关键决策

- **不显示步骤指示器**:玩家不感知 "第几步 / 共几步",只感受擦着擦着画面变干净的爽感
- **进度条显示全关合成进度**:3 步关每步占 33%,第 1 步擦到 50% → 总进度 16.7%
- **工具手动选择,选错有红色 toast 提示**:`工具不对哦`,1.5s 淡出
- **不做震动反馈**:超休闲品类反馈以视觉为主,震动放在后续版本
- **当前步骤所需工具用金色脉冲描边引导**:玩家通过试错+视觉提示自然学会切换工具
- **不做关卡选择网格**:首发只支持线性"下一关",留到 v1.1

### 5.3 胜利弹窗

- 半透明遮罩 + 白色圆角卡片
- 标题:`清洁完成!`
- 副标题:`焕然一新`
- 按钮:
  - 普通关:`再玩一次` / `下一关`
  - 末关(第 20 关):`再玩一次` / `分享`
- 普通关玩家如要分享,使用微信右上角"…"系统转发(`wx.onShareAppMessage` 已注册)

### 5.4 字号/间距常量

集中放在 `UI.js` 顶部,避免 magic number 散落,**不引入设计系统**。

## 6. 数据持久化(`src/storage.js`)

### 6.1 接口

```js
const KEY = 'clean_mini_progress'

export function loadProgress()
// 返回 { unlockedLevel: number, completedLevels: number[], lastPlayedAt: number }
// 首次返回 { unlockedLevel: 1, completedLevels: [], lastPlayedAt: 0 }

export function saveProgress(progress)

export function markLevelComplete(levelId)
// 写入 completedLevels;若是新关则 unlockedLevel = levelId + 1

export function reset()
// 调试用:清空进度
```

### 6.2 字段说明

| 字段 | 存 | 用途 |
|---|---|---|
| `unlockedLevel` | ✅ | 启动时直接进入"下一关该打的那关" |
| `completedLevels: []` | ✅ | 分享卡片显示"我清洁了 N 件物品" |
| `lastPlayedAt` | ✅ | 给运营钩子(7 天回流提醒等)预留,MVP 不用 |
| 当前关擦除百分比 | ❌ | 中途退出不保存,回来重头擦该关 |
| 单关耗时 | ❌ | 走 tracker 上报到云端,本地不存 |

### 6.3 启动流程

```
game.js → GameScene 构造 → loadProgress() → 直接进入 unlockedLevel 那一关
```

**没有主菜单**,符合超休闲零等待原则。

## 7. 埋点(`src/tracker.js`)

### 7.1 云开发准备

- 微信开发者工具开通云开发环境
- 创建集合 `events`,字段:`{ name, payload, openid, ts }`(`openid` 由云函数自动注入)
- 写云函数 `trackEvent`,只做一件事:把入参写入 `events` 集合
- 客户端调用:`wx.cloud.callFunction({ name: 'trackEvent', data: { name, payload } })`

### 7.2 接口

```js
// src/tracker.js
export function init() { wx.cloud.init({ env: 'xxx' }) }

export function track(name, payload = {}) {
  // 异步上报,不阻塞主流程,失败静默(.catch(() => {}))
}
```

### 7.3 事件清单

| 事件 | 触发时机 | payload |
|---|---|---|
| `app_launch` | `GameScene` 构造时 | `{ unlockedLevel }` |
| `level_start` | 进入关卡 | `{ levelId, category }` |
| `level_complete` | 关卡通关 | `{ levelId, durationMs }` |
| `level_quit` | `wx.onHide` 中途退出 | `{ levelId, progress, durationMs }` |
| `step_complete` | 步骤擦到阈值 | `{ levelId, stepIndex, durationMs }` |
| `wrong_tool` | 选错工具 | `{ levelId, stepIndex, expected, got }` |
| `share_click` | 玩家点分享 | `{ levelId, completedCount }` |
| `replay_click` | 玩家点再玩一次 | `{ levelId }` |

### 7.4 关键指标推导

- **次日留存**:`app_launch` 的 openid 第二天是否再出现
- **单局时长**:`level_complete - level_start`
- **关卡通过率**:`level_complete / level_start` 按 `levelId` 聚合(找卡关点)
- **卡哪一步**:`step_complete` 缺失分布
- **工具引导是否清晰**:`wrong_tool` 频次按 `levelId` 聚合
- **分享率**:`share_click / level_complete`(末关为主)
- **流失断点**:`level_quit` 按 `levelId` 聚合

### 7.5 容错

- 上报失败静默,不影响游戏体验
- **不做客户端缓存重传**:MVP 阶段够用,丢失 1% 事件不影响判断

## 8. 分享(`src/share.js`)

### 8.1 入口

1. 胜利弹窗的分享按钮(玩家主动)
2. 微信右上角"…"的转发(被动,需 `wx.showShareMenu` 注册)

### 8.2 接口

```js
import { loadProgress } from './storage.js'

export function setupSystemShare() {
  wx.showShareMenu({ withShareTicket: false })
  wx.onShareAppMessage(() => buildShareCard())
}

export function userShare() {
  wx.shareAppMessage(buildShareCard())
  // 上报埋点由调用方做(GameScene 在按钮点击时 track('share_click', ...))
}

function buildShareCard() {
  const completedCount = loadProgress().completedLevels.length
  return {
    title: completedCount > 0
      ? `我已清洁了 ${completedCount} 件物品,你来挑战?`
      : `太治愈了!来一起清理吧`,
    imageUrl: 'images/share/cover.png'
  }
}
```

### 8.3 分享卡片图

- 微信要求 5:4 比例,推荐 500×400
- MVP 阶段一张通用占位图,字段已留好,后期可按当前关卡换主题图

### 8.4 不做的事

| 项 | 理由 |
|---|---|
| 群排行 | 需要后端,与"无后端 MVP"冲突 |
| 分享得奖励 | MVP 不变现,无激励系统 |
| 动态合成分享卡片图 | `wx.canvasToTempFilePath` 复杂度高,等数据验证后再做 |

## 11. 测试策略

- **单元测试跳过**:游戏循环和触摸事件无传统单测价值,且会拖慢迭代
- **联调验证**:每个任务在微信开发者工具里手动点过 + 真机点过,作为完成标准
- **数据验证**:任务 5 完成后,在云开发控制台 `events` 集合里能看到自己点击产生的事件,作为埋点正确性证据

## 12. 风险与对策

| 风险 | 对策 |
|---|---|
| `getImageData` 在低端机卡顿 | 已节流到 500ms,扫描区域限定 `shoeRect`;如仍卡降级 1000ms |
| 占位图导致测试用户感知"很糙",影响留存数据判断 | 在埋点中增加 `app_version`,占位图版本与正式美术版本分开看留存 |
| 云开发免费额度不够(月调用 2 万次) | MVP 早期玩家量小够用;如超量,按 openid 抽样 50% 上报 |
| 微信审核拒绝(玩法单一/资源粗糙) | 占位图阶段不提审,内部体验码测试;正式美术版本再提审 |
